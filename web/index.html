About FLASHVIS

FLASHVIS runs real embedded filesystems, compiled to WebAssembly, against a JavaScript emulation of an SPI NOR flash chip. Every read, program, and erase the filesystem issues hits the emulated chip under real NOR rules and lights up the die, so you can watch and compare how filesystems behave on real flash.

What you are looking at

A small SPI NOR chip

The die is a 256 KB NOR region: 64 sectors of 4 KB, each sector 16 pages of 256 bytes. NOR has one governing asymmetry.

  • Programming clears bits from 1 to 0 at byte granularity. You can keep programming into blank 0xFF space without erasing, because writing 0xFF clears nothing.
  • To set bits back to 1 you must erase, and erase only works on a whole 4 KB sector. It is the slow, wearing operation.

Each cell on the die is one 256 byte page, and its fill height is the bytes programmed into it. Because programming only clears bits in blank 0xFF space, appending to a file grows it into that last page's slack in place, with no erase.

Updates never overwrite in place. A filesystem writes a fresh copy and marks the old one obsolete. When free pages run low, garbage collection reclaims a sector: it relocates the live data out, erases the sector, and adds one to that sector's erase count. Turn on the wear heatmap to tint each sector by that erase count, and hotspots pool wherever garbage collection keeps landing.

The timing preset is an ESP32-S3: a 256 byte read takes about 0.1 ms, programming a page about 1.5 ms, and erasing a sector about 21 ms. Erase is by far the most expensive operation, which is why filesystems work so hard to avoid it. The speed control in the header scales this simulated flash time up or down.

How each filesystem copes

FASTFFS and LittleFS

FLASHVIS compares two filesystems today, FASTFFS and LittleFS, with more planned. Both write copy-on-write and never overwrite in place, but they reclaim the freed space differently.

FASTFFS writes file data copy-on-write and commits each change with an append-only index record, leaving old versions as obsolete data to reclaim later. Its garbage collector is incremental, so it runs opportunistically in small steps in the background between file operations, which keeps reclaimable garbage low for a small, steady cost. The background GC control in the workload strip sets how much of that runs; with it on, FASTFFS holds garbage down, so the 50% default can undersell it, and what remains is reclaimed inline when free space runs tight.

LittleFS is also copy-on-write, with dynamic wear-leveling built in, but it reclaims space as part of its writes rather than in a separate collectible pass, so it has no incremental GC step for the visualizer to drive. That is why the background GC control does not appear when LittleFS is focused: the control governs the shared workload's GC share, which only a filesystem with incremental GC acts on.

This is a primer, not a benchmark. FASTFFS pays a periodic GC cost while LittleFS folds reclamation into its writes, and the die and wear heatmap let you watch each approach play out.
